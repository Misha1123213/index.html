# MET Академия: от сайта для TTK к платформе обучения

## 1. Что уже сделано (и это сильно)

Приложение уже обладает большим количеством «платформенных» механик, которые Duolingo сделал стандартом:

- **Адаптивная подача**: логический, цветовой и пространственный форматы вопросов; система выбирает формат на основе успешности.
- **Разные типы вопросов**: `select` (выбор состава), `missing` (чего не хватает), `truefalse` (верно/неверно), `color_fill` (цветовые группы), граммовки для напитков.
- **Spaced repetition / Item Strength**: каждый объект получает оценку силы `0-5`, по ней строятся «слабые места».
- **Умная репетиция**: ошибки сразу вставляют дополнительные вопросы в текущую сессию, а после урока — фаза remediation и consolidation.
- **Gamification**: XP, серии дней, ежедневная цель, жизни, достижения, аватар, рейтинг, конфетти.
- **Справочник**: expandable-список всех объектов с составом.
- **Статистика обучения**: сильная/слабая сторона, точность по категориям, динамика, прогресс по разделам.
- **Темы и настройки**: тёмная/светлая тема, звук, вибрация, анимации.

**Концептуально самое важное**: дата-модель уже почти универсальна — `объект` → `список компонентов` (+ опционально `граммовки` и `дистракторы`). Она покрывает блюда, десерты, напитки и полуфабрикаты.

## 2. Что сейчас ограничивает рост до платформы

### 2.1. Архитектура
- **Один монолитный `index.html` (4606 строк)**. Содержит вёрстку, стили, state, генерацию вопросов, логику, аналитику и рендеринг. Добавлять новые форматы и разделы всё сложнее.
- **Данные в JSON-файлах**. `dishes.json`, `drinks.json`, `desserts.json`, `drink_pf.json`, `kitchen_pf.json` — это контент, а не структура. Добавить новый курс или раздел = править файлы и код одновременно.
- **Всё в `localStorage`**. Прогресс, профиль, настройки не синхронизируются между устройствами, не восстанавливаются при смене телефона.
- **Нет бэкенда / авторизации**. Нет пользователей, нет админки, нет загрузки контента.

### 2.2. Контент
- **Формат жёстко завязан на кухню/напитки**. Нет generic сущности «объект + компоненты», которая позволила бы быстро добавлять другие домены.
- **Картинки привязаны к имени файла**. Почти нет fallback для отсутствующих фото; часть позиций без `image`.
- **kitchen_pf.json содержит много заглушек** (`— заполнить —`). Контент полуфабрикатов не завершён.

### 2.3. Пользовательский опыт
- **Нет понятия «курс/навык/уровень»** выше раздела. Duolingo-дерево есть (путь из нод), но нет юнитов, разблокировок по темам, босс-уровней.
- **Нет истории повторения**. Слабые места есть, но нет календаря spaced repetition (как Anki / Duolingo).
- **Рейтинг локальный и случайный**. Боты с рандомным XP; не мотивирует.
- **Нет социальных механик**: друзья, челленджи, shared decks, комментарии.

## 3. Видение платформы

> **Платформа для изучения составов любых объектов через микро-взаимодействия.**

Каждый объект = что-то, что имеет составляющие:
- **Кулинария**: блюдо → ингредиенты; напиток → ингредиенты + граммовки + техника; десерт → слои/компоненты.
- **Профессиональное**: коктейль → спиртные/миксеры/декор; кофе → зерно/молоко/пропорции/температура.
- **За пределами еды**: парфюмерные ноты, компоненты ПК, лекарственные препараты, строительные материалы, языковые конструкции (в Duolingo — «объект» = предложение, «компоненты» = слова).

Формат обучения остаётся тем же: **короткие сессии, разные форматы подачи, адаптация под слабые места, геймификация**.

## 4. Стратегический роадмап

### Фаза 1. Фундамент (чтобы перестать быть «сайтом для TTK»)

Цель: сделать приложение масштабируемым и generic, не теряя текущих данных.

1. **Универсальная дата-модель**
   - `Course` → `Unit` → `Lesson` → `Item` (объект) → `Component` (компонент).
   - `Item` имеет: `name`, `image`, `category`, `components[]`, `optionalMeta` (граммовки, техника, порядок, оборудование).
   - Дистракторы генерировать автоматически из пула компонентов курса, а не хранить вручную.

2. **Разделение приложения на модули**
   - Вынести state, data, question generators, renderers, analytics в отдельные ES-модули.
   - Внедрить Vite / Webpack для сборки и hot-reload.
   - Стили разбить по компонентам.

3. **Перевод JSON в editable-источник**
   - Создать `content/` папку с `courses.json` и yaml/markdown описанием курсов.
   - Сделать CLI/скрипт, который из `dishes.json` и др. генерирует новую универсальную схему.
   - Поддержать авто-загрузку картинок и fallback-изображения.

4. **PWA**
   - Service Worker, офлайн-режим, manifest, push-напоминания о ежедневной цели.

5. **Улучшение ввода граммовок**
   - Сделать ввод граммовок более удобным: слайдер, tolerance ±2г, визуальный «шкала».

### Фаза 2. Платформа обучения (core learning)

Цель: сделать обучение действительно платформенным, как Duolingo.

1. **Курс-дерево и навыки**
   - Визуальное дерево юнитов (не только линейный путь).
   - Разблокировка юнитов по условиям: пройти N уроков, набрать X XP, закрепить слабые места.
   - Босс-уровень: смешанные вопросы по всему юниту.

2. **Новые форматы вопросов**
   - `sequence` — порядок добавления/сборки компонентов.
   - `match` — связать компонент с объектом.
   - `equipment` — выбрать инвентарь/посуду для блюда.
   - `substitute` — чем заменить отсутствующий компонент.
   - `listen/write` (TTS) — для названий компонентов.
   - `image-to-component` — по фото/схеме выбрать части.

3. **Полноценный spaced repetition**
   - Календарь повторения с алгоритмом SM-2 / FSRS.
   - «Вкладка Практика» с очередью на повторение, разбитой на «горящие», «слабые», «новые».

4. **Профили и синхронизация**
   - Бэкенд (Firebase / Supabase / собственный): пользователи, авторизация, облачный прогресс.
   - Мультиплатформенность: web + мобильное приложение (Capacitor / PWA).

5. **Система уровней и навыков**
   - Уровни мастерства по категориям: «Знаю состав», «Знаю порядок», «Знаю граммовки».
   - Skill tree для разных компетенций (кухня, бар, десерты, п/ф).

### Фаза 3. Социальная платформа и контент

Цель: превратить приложение в место, где создают и делятся курсами.

1. **Реальный рейтинг и друзья**
   - Добавление по username, недельные челленджи, лиги.
   - Реальные друзья вместо ботов.

2. **Редактор курсов (авторинг)**
   - Конструктор курса: создать объект, добавить компоненты, выбрать форматы вопросов.
   - Импорт из CSV / Notion / JSON.
   - Предпросмотр урока и валидация данных.

3. **Каталог курсов**
   - Публичные курсы: барные напитки, кофе, вина, десерты, и т.д.
   - Рейтинг, отзывы, теги, поиск.

4. **Монетизация / организации**
   - Приватные курсы для ресторанов/сетей.
   - Группы пользователей, админ-панель, аналитика по сотрудникам.

5. **AI-помощник**
   - Генерация дистракторов, подсказок, объяснений ошибок.
   - Автоматическое создание вопросов из ТТК/меню.

## 5. Первые конкретные шаги (приоритет)

Чтобы сразу двигаться в сторону платформы, рекомендую следующий порядок:

1. **Нормализовать данные** (1-2 дня)
   - Написать скрипт, который превращает `dishes.json`/`drinks.json`/`desserts.json`/`kitchen_pf.json`/`drink_pf.json` в единый `courses.json`.
   - Убрать дублирование ингредиентов, заглушки, привязать изображения.

2. **Распилить `index.html` на модули** (3-5 дней)
   - `state.js`, `data.js`, `renderers/`, `generators/`, `analytics.js`, `ui.js`.
   - Подключить Vite для сборки.

3. **Сделать generic Course/Unit/Lesson** (2-3 дня)
   - Экран выбора курса (не только кухня/десерты/напитки).
   - Визуальное дерево юнитов.

4. **Добавить 2-3 новых формата вопросов** (2-3 дня)
   - `sequence` (порядок сборки) и `match` (связь компонент-объект) — сильно расширяют «уникальный формат подачи».

5. **Прикрутить бэкенд для прогресса** (3-5 дней)
   - Supabase/Firebase — минимум: auth + облачный прогресс.

6. **PWA + push-напоминания** (1-2 дня)
   - Сильно увеличивает retention.

## 6. Технологические рекомендации

| Слой | Сейчас | Рекомендация |
|------|--------|--------------|
| Frontend | Vanilla JS, monolith HTML | Vite + ES Modules + Web Components / (можно остаться на Vanilla, но разбить на модули) |
| State | global `state` object | Zustand / Pinia / собственный Store |
| Хранилище | `localStorage` | Supabase / Firebase / PostgreSQL + REST/GraphQL |
| Контент | JSON-файлы | CMS-структура: `courses.json` + редактор курсов |
| Сборка | нет | Vite + PWA plugin |
| Мобильное | Web | PWA → Capacitor / React Native / Flutter |
| AI | нет | OpenAI/Claude API для генерации вопросов и подсказок |

## 7. Риски и что делать с ними

- **Данные**: `kitchen_pf.json` сейчас неполный. Перед переходом на generic-модель нужно дозаполнить или пометить `draft: true`.
- **Пользовательская база**: сейчас нет auth. Синхронизация прогресса — первый шаг к бэкенду.
- **Контент**: без редактора курсов платформа не сможет масштабироваться. Но редактор можно сделать позже, после унификации модели.
- **Большой рефакторинг**: `index.html` 4606 строк. Начинать с модуля `data.js` и `generators.js`, а не с полной переписи.

## 8. Краткий вывод

У приложения уже есть отличная обучающая «начинка»: адаптивные форматы, геймификация, репетиция ошибок, статистика. Главное, что мешает назвать его платформой — монолитная архитектура, отсутствие generic дата-модели и отсутствие бэкенда/авторинга.

**Ближайшая большая победа**: унифицировать дата-модель в `объект + компоненты`, разделить `index.html` на модули и добавить экран выбора курсов. Это превратит «сайт для TTK» в прототип платформы, не отбрасывая уже сделанный контент.

Дальше — добавить новые форматы вопросов, бэкенд, PWA и редактор курсов.
